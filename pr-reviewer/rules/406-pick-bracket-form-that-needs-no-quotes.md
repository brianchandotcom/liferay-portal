# 406: Pick the Bracket Form That Needs No Quotes

In a shell script, write a conditional test as `[ ... ]` when its operands can stand without quotes, and as `[[ ... ]]` when the alternative `[` form would need quoted operands. Use `[` for file tests on paths you control (`[ -d ${pr_subdir} ]`), for integer comparisons on counters (`[ ${pid} -ne $$ ]`), and any other test whose operands cannot be empty, contain whitespace, or hold a glob character. Use `[[` for string compares on a variable that might be empty (`[[ -z ${state} ]]`, `[[ ${command} == check ]]`), for pattern matches (`[[ ${path} == *.diff ]]`), and for `=~` regexes — `[[` is a bash keyword that does not word-split or glob-expand its operands, so unquoted variable references stay safe even when the value is empty or contains spaces. Never combine `[` with quoted operands: write `[[ $(readlink ...) = ${PWD} ]]`, not `[ "$(readlink ...)" = "${PWD}" ]`.

**Rationale:** The POSIX `[` form reads as a plain command and matches the rest of the script's command syntax. The bash `[[` form is a keyword with its own parsing rules — no word splitting and no glob expansion on operands — which removes the quoting that `[` would otherwise require for any value that might be empty or hold whitespace. Writing `[ "${var}" = "value" ]` mixes the POSIX command with the quote noise that `[[` is designed to eliminate; the same test reads `[[ ${var} = value ]]` with `[[` and stays just as safe. Choosing between the two forms by the operand's quoting need keeps every test as short as it can correctly be.

A violation is `[ "${var}" ... ]` (the `[` form with quoted operands) where `[[ ${var} ... ]]` carries the same meaning unquoted, or `[[ ${var} ... ]]` on operands that need no quotes and would read more directly as `[ ${var} ... ]`.
